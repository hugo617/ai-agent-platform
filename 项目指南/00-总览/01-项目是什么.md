## 一句话定位

**这是一个「多租户 AI 智能体 SaaS 平台」的脚手架。**

拆开来说,它解决了三个核心问题:

1. **多租户** —— 多个企业客户共用一套系统,但数据严格隔离。A 公司绝对看不到 B 公司
   的数据。
2. **AI 智能体(Agent)** —— 内置了一个能用工具、能流式对话的 AI 助手(类似 ChatGPT
   那种体验)。
3. **SaaS** —— 它是一个「软件即服务」的底座,意味着有完整的用户/角色/权限体系,可以
   直接拿去做新产品。

> 💡 **什么是[租户](../附录/术语表.md#租户tenant)?** 简单理解成「一个企业客户」。
> 比如你做一个协作工具,每个注册的公司就是一个租户,公司里的员工是租户下的用户。

---

## 它能用来做什么?

这个项目本身是个 **MVP 级脚手架**,目标是:**你做新的 SaaS 产品时,不用从零搭用户系统、
权限系统、租户隔离——直接基于它改就行。**

举例,你可以基于它快速做出:
- 企业内部的知识库问答助手(用户登录 → 选租户 → 跟 AI 对话)
- 多团队的 AI 编程助手(不同团队数据隔离 + 按角色控制谁能用)
- 任何「需要账号体系 + 权限 + 可能接入 AI」的 SaaS 产品

## 两种登录方式(认证)

这个项目支持**两套**认证,可以按需选用:

| 方式 | 场景 | 怎么工作 |
|------|------|---------|
| **本地账号密码** | 快速开发、自建账号体系 | 用户名/密码 → 后端用 bcrypt 校验 → 发一个 JWT |
| **Logto(OIDC)** | 企业级、对接单点登录 | 把登录交给专业的 Logto 服务,后端只验证它发的 token |

> 💡 **关键设计**:无论哪种方式登录,后端都用**同一条**「验证当前用户」的管线处理,
> 所以下层代码完全感觉不到差异。这个设计很巧妙,详见
> [05-认证体系](../02-后端架构/05-认证体系.md)。

## 三大设计哲学

理解这三个原则,能帮你少踩很多坑:

### 1. 严格分层,依赖单向

代码分四层:**Controller → Service → Repository → Model**。依赖只能从上往下,不能回头。

```
api(路由)  →  services(业务)  →  repositories(数据)  →  models(表)
     ↑                                                              │
     └──────────────── 返回数据 ←──────────────────────────────────┘
```

**好处**:改某一层不会污染别的层;每层职责清晰,好测试。

> 详见 [01-分层架构与依赖方向](../02-后端架构/01-分层架构与依赖方向.md)。

### 2. 多租户隔离「一处强制,无法遗忘」

所有租户相关的表都带 `tenant_id`。**关键在于**:隔离逻辑统一写在 Repository 基类里,
查询时自动加 `WHERE tenant_id = 当前租户`。这样开发者写业务代码时**不可能**忘记加过滤。

> 详见 [04-多租户隔离](../02-后端架构/04-多租户隔离.md)。

### 3. 权限「声明式 + 双重校验」

- **声明式**:哪个接口需要什么权限,直接在路由上「声明」(像贴标签),不用在业务代码里
  到处写 `if 有权限:`。
- **双重校验**:HTTP 接口层校验一次;AI 工具被调用时,在工具内部再校验一次。这样即使
  AI 试图调不该用的工具,也会被拦住。

> 详见 [06-权限模型RBAC](../02-后端架构/06-权限模型RBAC.md)。

## 项目目前处于什么阶段?

按 README 的说法,这是 **Phase 1**:后端骨架 + 多租户权限 + 最简 AI Agent,前后端分离。

已完成的核心能力:
- ✅ 用户管理(完整的增删改查 + 分页搜索筛选)
- ✅ 多租户 + RBAC 三档角色(owner / admin / member)
- ✅ 组织架构树
- ✅ AI Agent + SSE 流式对话
- ✅ 审计日志、登录会话管理

后续计划(尚未做):
- 🔲 RAG 向量检索(pgvector 已就位)
- 🔲 Agent 持久化记忆
- 🔲 计费、配额
- 🔲 生产部署(K8s)
