## 📥 渐进式喂入原则(核心)

**不要一次性读完所有文档。** 按任务需要,分批读取。这样既省 token,又能聚焦。

### 三步法

1. **第一步**:先读本文件 + [README.md](README.md),建立全局索引意识。
2. **第二步**:根据用户任务,从下表选 2-4 篇**最相关**的文档精读。
3. **第三步**:需要定位具体代码时,配合 CodeGraph 工具(见下文)精确查询。

## 🔍 CodeGraph 配合使用(强烈推荐)

本项目文档讲「**是什么、为什么**」;CodeGraph 讲「**代码在哪、谁调用谁**」。两者互补。

### 现状

- `codegraph` CLI 已安装(版本 1.2.0)。
- 索引数据(`.codegraph/codegraph.db`)**不入库**——仓库只提交了 `.codegraph/` 目录壳
  (含它自己的 `.gitignore`),所以新环境 / clone 后需要先重建索引。

### 一键启用

如果需要精确定位符号关系(比如「谁调用了 `get_current_user`」),先建立索引:

```bash
codegraph init   # 在项目根目录运行,(重建).codegraph/ 索引
```

启用后,你就可以用 `codegraph_explore` 这类工具,用自然语言或符号名查询:

- 「`require_permission` 在哪些路由用到了?」
- 「`TenantScopedRepository` 的所有子类?」
- 「从 HTTP 请求到数据库的调用链?」

### 何时用文档,何时用 CodeGraph?

| 场景 | 用哪个 |
|------|--------|
| 理解某个机制「为什么这么设计」 | 📄 文档 |
| 看某个概念的完整说明(如多租户) | 📄 文档 |
| 查「某个函数被谁调用」 | 🔍 CodeGraph |
| 查「某类有哪些子类/实现」 | 🔍 CodeGraph |
| 找「某个符号定义在哪个文件」 | 🔍 CodeGraph |

## 📐 代码引用规范

文档里引用代码时,本项目统一用**符号锚点**而非行号(行号会随代码变动失效):

✅ 正确格式:
> 多租户隔离在 `app/repositories/base.py` 的 `TenantScopedRepository` 类里实现。

❌ 避免:
> 多租户隔离在第 49 行。(`49` 行可能已经变了)

读文档遇到这种引用时,你可以直接用文件路径打开,或在 CodeGraph 启用后查询符号。

## ⚠️ 几个容易踩的坑

1. **三种 token 别搞混**:本地密码登录(HS256)、Logto OIDC(RS256)、开发 token(RS256)。
   详见 `02-后端架构/05-认证体系.md`。
2. **多租户隔离有两套**:普通实体用 `TenantScopedRepository`;User 这种租户关系在
   关联表里的,要自己 join 过滤。别只加一层。详见 `02-后端架构/04`。
3. **权限校验在两处**:Controller 层声明式校验(`require_permission`)+ Agent 工具内部
   再次校验。改权限时两处都要想到。
4. **测试用 SQLite 内存库**:生产是 PostgreSQL。少数语法/行为可能有差异(如部分索引)。

## 📝 你的工作守则

- **修改代码前**,先确认读过相关架构文档,理解现有约定(尤其是分层依赖方向)。
- **写新代码时**,参考同层现有模式(如新增 service 参考 `app/services/user_service.py`)。
- **解释给人类时**,可以引用文档路径让他们自己深入阅读。
- **不确定时**,用 CodeGraph 查实际的调用关系,别凭记忆猜。
