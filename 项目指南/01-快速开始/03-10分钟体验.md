## 🤖 第二站:Agents(AI 智能体)

点左侧菜单 **Agents**,进入 `/agents`。这是管理 AI 智能体的地方。

### 试着创建一个 Agent

1. 点「新建」按钮
2. 填名字(比如「我的助手」)和描述
3. 保存

**发生了什么?**
- 后端在 `agents` 表插入一条记录,**自动带上你当前租户的 `tenant_id`**
- 你作为 owner 有 `agents:create` 权限,所以能创建成功

> 💡 这里创建的 Agent 是「AI 助手的配置」,有了它才能开始 AI 对话。

### 试试和它对话(AI 对话)

Agent 详情或对话入口处,输入一句话(比如「你好,介绍一下你自己」),发送。

**你会看到**:文字**一段一段**冒出来,像 ChatGPT 那样。这就是 **SSE 流式响应**。

**背后发生了什么?**(详见 [07-Agent与LLM集成](../02-后端架构/07-Agent与LLM集成.md))
1. 前端发请求到 `/api/v1/chat/stream`
2. 后端用 LangGraph 构建一个 ReAct Agent,接上 OpenAI
3. 大模型一边想一边吐字,后端用 SSE 把每个字片段推给前端
4. 对话结束,完整回复存进数据库(对话记录)

> ⚠️ **没填 OpenAI key?** 这一步会报错。不影响体验其他功能。

## 👥 第三站:Users(用户管理)

点 **Users**,进入 `/users`。这是完整 CRUD 的用户管理页,也是项目里**最完整的范例**。

### 玩玩这些功能

| 功能 | 怎么试 | 背后的设计亮点 |
|------|--------|---------------|
| **搜索** | 顶部搜索框输入名字,回车 | 后端 `ilike` 模糊查询 |
| **筛选** | 用状态下拉框选「活跃/锁定」 | 后端按 status 过滤 |
| **排序** | 点排序下拉,按创建时间/名字 | 后端支持多字段排序 |
| **分页** | 翻到第 2 页 | 后端分页,前端 `<Pagination>` 组件 |
| **统计卡片** | 看顶部数字(总数/活跃/锁定/本月新增) | 单独的 `/users/statistics` 接口 |
| **新增用户** | 点新建,填表单 | 密码用 bcrypt 加密存储 |
| **改状态** | 行操作菜单 → 锁定/停用 | 改完立即生效 |
| **重置密码** | 行操作菜单 → 重置密码 | 给新密码,加密存储 |
| **删除** | 行操作菜单 → 删除 | **软删除**(数据还在,只是标记删除) |

> 💡 **重点观察**:每次操作后,右上角会弹 toast 提示成功/失败,表格自动刷新。
> 这是 TanStack Query 的功劳——详见
> [04-数据获取TanStackQuery](../03-前端架构/04-数据获取TanStackQuery.md)。

> 💡 **软删除** 是项目的惯例:删了的数据其实没真消失,只是 `is_deleted=True`。
> 这样能恢复、能审计。查询时自动过滤掉已删的。详见
> [03-数据库与ORM](../02-后端架构/03-数据库与ORM.md)。

## 🔐 第四站:Roles & Permissions(角色与权限)

点 **Roles** 和 **Permissions**,看看权限体系。

### 三档默认角色

每个租户创建时,系统自动「种」(seed)好三个角色:

| 角色 | 能做什么 |
|------|---------|
| **owner**(所有者) | 一切都能做(管 Agent、管用户、管角色、管组织) |
| **admin**(管理员) | 大部分能做,但不能删 Agent、不能管角色 |
| **member**(普通成员) | 只能看 Agent、能对话,不能管用户 |

> 💡 这些规则定义在 `app/services/permission_service.py` 的 `seed_tenant_defaults`。
> 详见 [06-权限模型RBAC](../02-后端架构/06-权限模型RBAC.md)。

### 试试权限隔离

1. 创建一个新用户,角色设为 **member**
2. (如果支持)用这个 member 登录,或者想象你是 member
3. 你会发现:member 看不到「用户管理」菜单,点用户接口会返回 **403 Forbidden**

**这就是 RBAC 在工作**——不同角色看到的世界不一样。

## 🏢 第五站:多租户(理解核心概念)

虽然界面切换租户的功能可能在后续阶段才完整,但**理解这个概念**很重要:

> **场景想象**:假设你的产品卖给两家公司——A 公司和 B 公司。
> - A 公司的 admin 登录,**只能看到** A 公司的 Agent、用户、对话。
> - B 公司的数据对 A 公司来说**完全不存在**,连查都查不到。
> - 反之亦然。

**怎么实现的?** 所有表都有 `tenant_id` 列。查询时,Repository 层**自动**加
`WHERE tenant_id = 当前用户的租户`。开发者写业务代码时**根本不用**手动加过滤,
所以「忘了加过滤导致数据泄露」这种事故几乎不可能发生。

> 详见 [04-多租户隔离](../02-后端架构/04-多租户隔离.md)。
