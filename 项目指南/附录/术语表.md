## A

### API
**应用程序接口**。这里指后端暴露的 HTTP 接口,前端通过它拿数据。
- 项目里:所有接口在 `app/api/v1/` 下,统一加 `/api/v1` 前缀。
- 例子:`GET /api/v1/users` 拿用户列表。

---

## B

### bcrypt
一种**密码哈希算法**。用户密码不能明文存数据库,bcrypt 把密码变成一串乱码,
即使数据库泄露也无法还原出原密码。
- 项目里:`app/core/password.py` 的 `hash_password` / `verify_password`。

---

## C

### Casbin / pycasbin
一个**权限引擎**。你把「谁能做什么」的规则告诉它,它帮你判断某个请求是否允许。
- 项目里:用 pycasbin 做 RBAC(基于角色的权限控制),规则定义在 `casbin_model.conf`。
- 详见 [06-权限模型RBAC](../02-后端架构/06-权限模型RBAC.md)。

### Controller 层(控制器层)
代码分层里的**第一层**,负责接收 HTTP 请求、做权限校验、调用 Service。
- 项目里:`app/api/v1/` 目录。
- 类比:餐厅的**服务员**,接单、核对身份。
- 详见 [01-分层架构](../02-后端架构/01-分层架构与依赖方向.md)。

### CORS(跨域)
浏览器安全机制:前端地址和后端地址不同源时,默认会拦截请求。后端要明确「允许」前端地址。
- 项目里:`app/main.py` 用 `CORSMiddleware`,`CORS_ORIGINS` 配置允许的地址。

---

## D

### DTO(Data Transfer Object,数据传输对象)
接口之间传递数据的「包裹」。项目里用 Pydantic 模型当 DTO。
- 项目里:`app/schemas/` 目录定义请求/响应的数据结构。

---

## J

### JWT(JSON Web Token)
一种**令牌格式**。用户登录后拿到一串 JWT,之后每次请求带上它证明身份。
- 项目里有三种 JWT:本地密码登录(HS256)、Logto(RS256)、开发 token(RS256)。
- 详见 [05-认证体系](../02-后端架构/05-认证体系.md)。

---

## L

### Logto
一个开源的**登录服务**(OIDC 提供商)。把登录、注册、第三方登录等专业活儿交给它,
你的后端只负责验证它发的 token。项目里是可选的,本地开发可以用账号密码登录代替。
- 详见 `docs/LOGTO_SETUP.md`。

---

## M

### Model 层(模型层)
代码分层里的**第四层**,定义数据库表长什么样(用 Python 类描述)。
- 项目里:`app/models/` 目录,用 SQLAlchemy ORM。
- 类比:食材的**清单**。

### MVP(Minimum Viable Product,最小可行产品)
只做最核心功能、能跑起来的版本。本项目定位是「MVP 级 SaaS 脚手架」——地基打好,
盖什么楼你自己定。

---

## O

### OIDC(OpenID Connect)
基于 OAuth 2.0 的**身份认证协议**。Logto 就是 OIDC 提供商。简单理解:一套「怎么
安全地登录」的标准。

### ORM(Object-Relational Mapping,对象关系映射)
用 Python 对象操作数据库,不用手写 SQL。比如 `user.name = "张三"` 就能改数据库。
- 项目里:用 SQLAlchemy 2.0(异步版)。

---

## P

### pgvector
PostgreSQL 的一个**扩展**,让数据库能存和搜索「向量」(用于 AI 相似度检索)。
- 项目里:docker-compose 用的 `pgvector/pgvector:pg16` 镜像自带。
- 用途:将来做 RAG(知识库检索)时用。

### Pydantic
一个 Python 库,用来**定义和校验数据结构**。比如规定「创建用户时 email 必填且格式
正确」,Pydantic 自动帮你校验。
- 项目里:`app/schemas/` 用它定义请求/响应结构,`app/core/config.py` 用它读配置。

---

## R

### RBAC(Role-Based Access Control,基于角色的访问控制)
一种权限管理思路:**不直接**给每个人配权限,而是「人 → 角色 → 权限」。
张三是 owner 角色,owner 角色有「删除用户」权限,所以张三能删用户。
- 详见 [06-权限模型RBAC](../02-后端架构/06-权限模型RBAC.md)。

### ReAct Agent
一种 AI 智能体的模式:Reason(思考)+ Act(行动)。AI 先想该做什么,调用工具,
看结果,再继续想。本项目用 LangGraph 的 `create_react_agent` 构建。
- 详见 [07-Agent与LLM集成](../02-后端架构/07-Agent与LLM集成.md)。

### Repository 层(仓库层)
代码分层里的**第三层**,只管读写数据库,不掺业务逻辑。**多租户隔离就在这层强制**。
- 项目里:`app/repositories/` 目录,基类在 `app/repositories/base.py`。
- 类比:餐厅的**仓库管理员**,只管取食材。

---

## S

### SaaS(Software as a Service,软件即服务)
一种软件交付模式:用户不用安装,通过浏览器/网络就用,通常按订阅收费。
本项目就是个 SaaS 的脚手架——支持多租户(多客户共用一套)。

### seed(种子数据)
系统初始化时自动写入的「默认数据」。比如创建租户时自动种好三个默认角色。
- 项目里:`app/services/permission_service.py` 的 `seed_tenant_defaults`。

### Service 层(服务层)
代码分层里的**第二层**,装着**业务逻辑**:组合多个数据操作、记日志、管事务。
- 项目里:`app/services/` 目录。
- 类比:餐厅的**厨师**,实际做菜。

### SSE(Server-Sent Events,服务器推送事件)
一种让服务器**主动**往浏览器推数据的技术。AI 对话时,字一个一个冒出来就靠它。
- 项目里:`app/api/v1/chat.py` 的 `chat_stream`,返回 `text/event-stream`。

### 软删除(Soft Delete)
删除时不真删数据,而是打个标记(`is_deleted=True`)。查询时自动过滤掉已删的。
- 好处:能恢复、能审计、不会因为外键关联导致删不掉。
- 详见 [03-数据库与ORM](../02-后端架构/03-数据库与ORM.md)。

---

## T

### tenant(租户)
**一个独立的客户/组织**。多租户系统里,每个租户的数据互相隔离。
- 类比:一栋写字楼里,每家公司是一个租户,共用电梯但各自的办公室互不干扰。
- 项目里:每个租户有独立的 `tenant_id`,所有数据按它隔离。

### Tenant-Scoped(租户隔离的)
「带租户归属的」数据。这类表都有 `tenant_id` 列,用 `TenantScopedRepository` 查询。

## 虚拟环境(venv)

Python 的「**独立小房间**」。在 `.venv/` 目录里装依赖,不会污染系统 Python。
- 创建:`python3 -m venv .venv`
- 激活(macOS/Linux):`source .venv/bin/activate`
- 项目里:`.venv/` 目录(已在 `.gitignore`,不提交)。
