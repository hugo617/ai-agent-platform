## 2. 共识问题(2+ 评审者独立指出,高置信度)

### 🔴 共识严重问题(必修)

#### C-1. §8 阶段 2 步骤顺序与 §13 风险声明自相矛盾 [opus + sonnet 独立指出]

- **位置**: §8 阶段 2「步骤 1. 拆 AGENTS.md」先于「步骤 2-5. 新建目标文档」
- **冲突**: §13 倒数第 3 条明确声明「拆出去的内容必须**在新位置存在后**才能从 AGENTS.md 删除(顺序依赖)」,但实施步骤正好反了
- **后果**: 阶段 2 中间任何一步失败 → AGENTS.md 出现断链窗口
- **实测核实**: `harness/docs/doc-impact-assessment.md` 当前不存在,断链会真实发生
- **必修改法**: 把阶段 2 重排为「**2a. 先建 4 份新文档** → **2b. 最后一步才编辑 AGENTS.md 移出冗余段**」

#### C-2. §6 多模型投票机制在当前环境无法真异构 [opus + haiku 独立指出]

- **位置**: §6.1.2/§6.1.3 假设 3 个异构模型(GLM/Claude/GPT) vs §13.3 自承「当前环境仅 GLM 系列」
- **冲突**: 投票机制核心前提(异构模型族)在当前环境跑不起来;要么退化成 GLM 三次自评(共谋风险),要么依赖用户外接(成本+token 流转)
- **后果**: 验收标准 §10.7「multi-model-voting.md 含双模式」能过(文档可写),但机制不可用
- **必修改法**:
  - **方案 A(opus 建议)**: §1.3 决策表显式标注「投票机制为未来态,当前仅文档化定义 + 自举评审用同模型多 agent 模拟」;§6 增加「降级方案:单模型多 agent + 不同 system prompt 制造独立性」
  - **方案 B(haiku 建议)**: 把 §6 从「双模式同时做」降级为「**仅文档化机制 + 单任务试点**」,删掉 §12 强制自举,改为「plan 用单模型评审过即可启动;§6 机制落地后挑下一个真复杂任务试点,试点结论回写 §6」

#### C-3. §12 自举评审存在逻辑悖论 [haiku 主指, opus 隐含]

- **位置**: §12 让 plan 自身用「多模型投票机制」评审(自举)
- **问题**: 机制本身未经验证就用机制验证自己——第一份被评审的产出物是机制本身,而非用机制评审的产出物
- **本次实操验证**: 本次评审用的就是 GLM 系列的 3 个不同尺寸(opus/sonnet/haiku 实为同家族 GLM-4.6/4.5/4.4 模拟异构),**正好印证了 haiku 的「共谋风险」担忧**——3 票都在「可执行性」维度给了 1 分,这种高度一致性既是真问题也是共谋信号
- **必修改法**: 采用 haiku 方案 B,把 §6 从 plan 主线降级为独立后续任务,本次 plan 只做 §3-5+7

### 🟡 共识中等问题(建议改)

#### C-4. §5 Hook 配置应改用 workspace 级 [sonnet + haiku 独立指出, opus 轻微提]

- **位置**: §5.1 配在用户级 `~/.zcode/cli/config.json`
- **问题**: 影响所有 ZCode 项目;其他项目没有 `scripts/skill-counter.sh` 时每次 Skill 调用都失败(日志噪音 + 误导)
- **改法**: 改用 workspace 级 `<repo>/.zcode/config.json`,仅本仓库生效 + 可入库

#### C-5. §5.2 skill-counter.sh 脚本有 3 处可执行性隐患 [sonnet 主指 + opus 部分认同]

1. **注释/代码字段名不一致**: 注释写 `tool_input.skill_name`,代码取 `tool_input.skill`
2. **heredoc 变量注入脆弱**: `<<PY` 未加引号,路径含中文/空格时易破坏 Python 字面量
3. **stdout 陷阱**: 调试期 echo 会触发 hook 输出 schema 校验失败(diagnosing-hooks 陷阱 8)
- **改法**: 字段名先对齐待实测;heredoc 改 `<<'PY'` + 环境变量传参;所有日志走 stderr

#### C-6. §3.1 引用的 doc-impact-assessment.md 在 §9 影响面清单漏列 [sonnet 主指 + opus 实测确认]

- **位置**: §3.1 vs §9
- **问题**: §3.1 说要新建此文件,但 §9「文档新建 5 份」明细没有它
- **改法**: §9 改为 6 份并补上,或明确合入 task-workflow.md

## 3. 独立问题(单评审者指出,需主会话判断)

### opus 独立指出

- **O-1[中]**: §13 风险表漏了 diagnosing-hooks 明确列为高频陷阱的 2 项——`timeout` 单位(command 是秒 vs process 是毫秒) + JSON 输出严格 schema
- **O-2[中]**: §6.1.4 rubric 与 §12.3 评审 rubric 是两套不同维度但都叫「6 维度 rubric」,易混淆
- **O-3[轻]**: §3.1 目标「~85 行」与 §10 验收「≤100 行」不一致
- **O-4[轻]**: §7.7 引用「sess_f122bde8 教训」但无摘要,外部评审者无法核对

### sonnet 独立指出

- **S-1[轻]**: §5.1 `timeoutMs: 3000` 用于 `type: command` hook 易踩单位 pitfall,建议改 `timeout: 3`(秒)
- **S-2[中]**: §4.2 harness-router `disable-model-invocation: true` 与 §7.4「harness-router → /grill-with-docs」语义有张力(前者只能用户手动调,后者暗示 agent 自动调)
- **S-3[轻]**: §10.8「零外部 CSS 依赖」应补「Mermaid JS 走 4 源 fallback,离线时降级为代码块」
- **S-4[轻]**: §7.7 Tailwind 预编译假设本机有 Node+npx,本仓库根目录无 package.json,需在 /tmp 临时目录跑

### haiku 独立指出

- **H-1[严重]**: §7 HTML 花费 4-6 小时做 80KB 自包含文档,**ROI 存疑**——markdown 已经是 agent 和人类的一等公民,HTML 是「展示文档」不进入 agent 工作流。建议降级为可选项,或只做 mermaid 主流程图塞进 task-workflow.md
- **H-2[中]**: §3.3 引入 `bug-` id 前缀与 feature_list.json 现有约定未核实,可能破坏 init.sh 校验
- **H-3[中]**: §8 阶段 4「不实施 skill 封装」与 §6 机制断层——投票机制靠 agent 自觉读文档执行,**重复了 §1.1 缺口 5 的同一个痛点**
- **H-4[轻]**: 本次新增 markdown 接近 650 行,对一个「瘦身」任务总体积反而在涨,建议给每份文档设硬性行数上限
- **H-5[轻]**: §7.6 Skill 字典 + §4.3 路由表 + §3.1.1 AGENTS.md 路由表三处重复描述 skill 触发关系,建议以 AGENTS.md 路由表为单一真相源

### 4.2 共谋信号识别(本次评审的元发现)

**关键观察**: 3 个评审者(实为同家族 GLM 不同尺寸模拟)在「可执行性」维度全部独立给出 1/2 分,且都指向 §8 阶段 2 顺序问题。这既证明**该问题真实存在**,也证明**异构模型缺失时共谋风险非虚**——3 票都受相似训练数据影响,对同一类问题有相同盲区。

**对 §6 机制的反馈**: 本次评审实操暴露了「单家族多 agent 模拟异构」的根本局限——无法提供真正的视角多样性。这强化了 haiku P1 的判断:**当前环境不适合把多模型投票作为主线机制**。

### 5.2 修订后重审路径

1. 作者按 §4.3 P0(5 条)+ P1(5 条)修订 plan
2. 修订版提交后,**重新触发评审**——但本次不再用「多模型投票自举」(因 §6 已降级)
3. 改用「单模型评审 + 用户最终拍板」(对齐 haiku 方案 B 的精神)
4. 重审通过后进入实施

### 5.3 对本次评审本身的元评估

本次评审作为「多模型投票机制」的**首次实操**,暴露了 3 个设计问题,这些应回写到未来的 `multi-model-voting.md`:

| 暴露的问题 | 回写到的位置 |
|---|---|
| 同家族多尺寸模型模拟异构 = 共谋风险非虚(3 票可执行性全 1 分既是真问题也是共谋信号) | §6.1.5 避坑设计补强 |
| 自举评审形成「未经验证的机制验证自己」的逻辑悖论 | §12 删除强制自举,改为「机制落地后挑真任务试点」 |
| rubric 维度过粗(0-2 分)导致 9-10 分都归 Accept/Revise 边界,缺区分度 | 未来考虑 0-3 分或加权重 |

**结论**: 多模型投票机制本身有价值,但**当前环境不具备实施条件**。本次评审的最大价值不是通过/否决 plan,而是**用实操验证了机制设计缺陷**——这比纸面设计更值钱。
