Harness 工程实践指南

这是 ai-agent-platform 项目的 Harness 工程总览。 本文档讲「工程实践是什么、为什么这么做、怎么用」， 配合 AGENTS.md（入口）+ harness/docs/（规则文档）+ skills（自动化）使用。

🎯
这套工程解决什么
AI 编码助手天然「顺手多改 / 半成品 / 假完成」。Harness 用 WIP=1 + 完成绑定证据 + 不越界 3 条硬规则约束它。
🛠️
由什么组成
5 件套：AGENTS.md（入口）+ feature_list.json（任务真相源）+ progress.md（日志）+ plan-*.md（步骤）+ skills（自动化）。
🚀
怎么用
按 ./init.sh → 读 progress.md → 读 feature_list.json 顺序开工；按路由表自动调 skill。
📌 阅读建议　 首次阅读按顺序（开篇 → 主流程 → 工作流导览 → 现状对比）； 日常使用直接跳到 工作流导览 或 Skill 字典 查阅。

📍 三个关键关卡
  1. 身份认证(get_current_user):你是谁?
  2. 权限校验(require_permission):能做这事吗?
  3. 租户隔离(Repository):只看你租户的数据。
这是后端请求的硬关卡,改权限时三处都要想到。
📍 三个反模式(必避)
  • 顺手多改 —— 不在当前任务范围的代码,哪怕看着再痒也别动。
  • 半成品标完成 —— status=passing 的唯一方式是验证真跑过 + 证据。
  • 脑子里的算数 —— 决策/进度/约定必须落到文件。
工作流导览

Agent 怎么走完一个任务(8 步)

这一段带领读者看 agent 实际怎么从开工到收尾。每张卡含 5 个字段：触发条件 / 读什么 / 调什么 skill / 产出什么 / 下一步判定。

1

会话开工

触发条件
每轮 ZCode 会话开始(用户打开仓库)
读什么
AGENTS.md → progress.md → feature_list.json
调什么 skill
无(直接跑 ./init.sh)
产出 / 下一步
确认起点干净(ruff + pytest 全绿) → 进步骤 2
2

新任务登记

触发条件
用户提新需求 / 选 priority 最高的 not_started
读什么
相关架构文档 + harness/docs/ 已有 plan 范例
调什么 skill
/harness-router(迷茫时)→ /grill-with-docs(烤清需求)
产出 / 下一步
共识 + CONTEXT.md(纸面轨迹) → 进步骤 3
3

落 PRD

触发条件
grill 共识达成,需求清晰
读什么
harness/docs/prd-template.md(PRD 强化模板)
调什么 skill
/to-spec(综合,不访谈)
产出 / 下一步
harness/docs/plan-<id>.md → 进步骤 4
4

拆切片

触发条件
PRD 完成,任务复杂(改 >5 文件 / 跨模块)
读什么
刚完成的 plan 文档
调什么 skill
/to-tickets(tracer-bullet 垂直切片 + blocking edges)
产出 / 下一步
N 个 ticket(每个独立可验证) → 进步骤 5
5

实施

触发条件
切片就绪,context 未到 60%
读什么
ticket 文件 + 相关架构文档(改前必读)
调什么 skill
/implement(内部驱动 /tdd 红绿循环)
产出 / 下一步
代码改动 + 测试 → 进步骤 6
6

Code Review

触发条件
实施完成,代码改完 + 自测过
读什么
代码 diff + 测试结果
调什么 skill
/code-review(Standards + Spec 双轴)。复杂任务可触发多模型投票[未来态]
产出 / 下一步
Pass / Revise / Block 结论 → Revise 回步骤 5,Pass 进步骤 7
7

关闭任务

触发条件
code-review Pass + ./init.sh 全绿
读什么
feature_list.json 当前任务条目
调什么 skill
无(手动填 evidence + 改 status=passing)
产出 / 下一步
任务状态 = passing + evidence 已填 → 进步骤 8
8

收尾

触发条件
会话即将结束(用户要离开 / context 满)
读什么
harness/clean-state-checklist.md
调什么 skill
手动对照清单打勾 + 更新 progress.md Session 记录
产出 / 下一步
干净状态留给下轮会话 → ✅ ship it / commit / PR

Skill 字典

8 张 skill 速查卡

L1 必会 5 张(harness-router / grill-with-docs / to-spec / to-tickets / implement)+ L1 评审 1 张(code-review)+ L2 进阶 2 张(tdd / handoff)。

🧭

/harness-router

L1 路由
任务状态路由器 —— 用户迷茫时手动求助。
触发: 用户键入 /harness-router(disable-model-invocation: true)
输入: 当前任务状态描述
输出: 推荐的下一步 skill
注意: agent 自动触发靠 AGENTS.md 路由表(硬触发),本 skill 是软辅助
🔥

/grill-with-docs

L1 必会
有 codebase 时烤清需求 —— 有状态访谈,保留成果到 CONTEXT.md。
触发: 新任务想法 + 有 codebase
输入: 模糊想法 + 仓库上下文
输出: 共识 + CONTEXT.md(纸面轨迹)
误用警告: 没 codebase 用 /grill-me(无状态版)
📝

/to-spec

L1 必会
把共识落成 PRD —— 不访谈,综合已有线索。模板见 prd-template.md。
触发: grill 共识达成
输入: CONTEXT.md + 已有讨论
输出: plan-<id>.md(含 Problem/Solution/User Stories/实施决策/测试决策/Out of Scope)
误用警告: 不要用来重新访谈,需求不清先回 /grill-with-docs
🎫

/to-tickets

L1 必会
拆 tracer-bullet 垂直切片 —— 每片切穿 schema→API→UI→test,声明 blocking edges。
触发: PRD 完成 + 任务复杂(改 >5 文件)
输入: plan-<id>.md
输出: N 个 ticket 文件(每个独立可验证)
例外: wide refactor 用 expand–contract 序列,不切片
🔨

/implement

L1 必会
基于 PRD + ticket 实施 —— 内部驱动 /tdd 红绿循环。
触发: ticket 就绪
输入: ticket 文件 + 架构文档(改前必读)
输出: 代码改动 + 测试
注意: 严守边界,不顺手改其他代码
🔍

/code-review

L1 评审
双轴评审(Standards + Spec)—— 复杂任务可触发多模型投票[未来态]。
触发: 实施完成 + 代码改完
输入: 代码 diff + 测试结果
输出: Pass / Revise / Block 结论
未来: 多模型投票就绪后,复杂任务自动升 3 模型
🚦

/tdd

L2 进阶
红绿循环 —— 先写失败的测试,再写代码让它过,再重构。
触发: /implement 内部驱动 / 单独使用
输入: 想要的具体行为
输出: 测试 + 实现 + 可选重构
独立用: 想测试驱动写某行为但不走完整 spec 时
📦

/handoff

L2 进阶
context 满 60% 时换新 session —— 压缩对话成 markdown 文件,新 session 引用。
触发: context 接近 60% / 需要分支(如切去 /prototype)
输入: 当前对话
输出: markdown 文件(在新 session 引用)
注意: 不在原地继续,而是新开 session;区别于 /compact(同对话总结)
未来态 · 待试点

📋 多模型投票机制

⚠️ 当前状态:未来态 · 待单任务试点验证
本机制已文档化于 harness/docs/multi-model-voting.md,不实施实操。 试点通过前,复杂任务评审仍用单模型 /code-review 双轴(Standards + Spec)。
模式 A:写方案合并取优
用于 /to-spec 输出
3 个异构模型各写方案
↓
主模型融合(不是投票,标注每段来源)
↓
最终 PRD
模式 B:评审多数票
用于 /code-review 完成度
3 个异构模型各评 Pass/Revise/Block + rubric 打分
↓
多数票(≥2 票一致)→ 最终结论
3 票各异 → 主模型 + rubric 仲裁(不凭感觉)
🚫 避坑设计(基于本 plan v1 评审实测)
🔴 共谋风险 —— 同家族模型相似训练数据导致相同盲区。必须异构家族(GLM/Claude/GPT),禁止同家族多尺寸模拟(本 plan v1 评审已验证此路不通)。
🔴 自举悖论 —— 未经验证的机制验证自己。首次应用必须评非机制本身的产出物。
🟡 rubric 区分度 —— 0-2 分让 9-10 分都卡 Accept/Revise 边界。未来考虑 0-3 分或加权。
🟡 溯源验证 —— 强制至少一票真读源码 / 官方文档核实,防模型编事实。
详见 harness/docs/multi-model-voting.md(190 行,含双模式定义 / 两套 rubric / 触发条件 / 避坑设计 / 试点记录)
自动化

两个自动化机制

📍 Skill 计数器(ZCode Hook)
每次 Skill 调用自动 +1 到 .skill-counters.json。
脚本: scripts/skill-counter.sh(入库)
配置: 用户级 ~/.zcode/cli/config.json(用户私有)
机制: PostToolUse hook,matcher ^Skill$
隔离: cwd 守卫(仅 ai-agent-platform 记录,其他项目静默退出)
原因: workspace hook 被 ZCode security policy 拦截,降级用户级(详见 hook-setup-guide.md)
🤖 自动触发路由表(AGENTS.md)
按任务状态变化硬触发 skill,不靠 agent 自觉。
位置: AGENTS.md §自动触发规则
双保险: 路由表(硬)+ /harness-router skill(软)
规则: 7 行路由(新增任务→grill-with-docs / 需求清楚→to-spec / 实施完成→code-review 等)
流程图: harness/docs/task-workflow.md §7(mermaid)
