## 1. 双模式

### 1.1 模式 A:写方案合并取优(用于 `/to-spec` 输出)

**输入**:`/grill-with-docs` 共识
**输出**:最终 PRD(标注每段来源 model-A/B/C)

```
grill 共识
  ├─ 模型 A(如 GLM-4.6) → 方案 v1
  ├─ 模型 B(如 Claude)   → 方案 v2
  └─ 模型 C(如 GPT)      → 方案 v3
       ↓
  主模型(agent 当前模型)→ 合并取优(融合,不是简单投票)
       ↓
  最终 PRD(每段标注来源 model-A/B/C)
```

**关键**:模式 A 是**融合**不是**投票**。主模型读 3 份方案,挑每段最优组合成新方案,标注来源便于追溯。

### 1.2 模式 B:评审多数票(用于 `/code-review` 完成度)

**输入**:实施完成的代码 diff
**输出**:Pass / Revise / Block + rubric 打分

```
实施完成的 diff
  ├─ 模型 A → Pass/Revise/Block + rubric 打分
  ├─ 模型 B → Pass/Revise/Block + rubric 打分
  └─ 模型 C → Pass/Revise/Block + rubric 打分
       ↓
  多数票(≥2 票一致)→ 最终结论
  分歧时(3 票各异)→ 主模型 + rubric 仲裁
```

**关键**:模式 B 是**投票**。3 票一致或 2:1 取多数;3 票各异时主模型按 rubric 仲裁(不能凭感觉)。

## 2. Rubric(两套,严格区分)

> 🚨 **plan 评审和 code 评审用不同的 rubric**(回应 review O-2)。两者都叫「6 维度 rubric」但维度不同,易混淆。本文档显式区分。

### 2.1 plan/方案评审 rubric

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| **正确性** | 有事实错误 | 小事实瑕疵 | 全对 |
| **完整性** | 漏关键交付物 | 漏非关键项 | 全 |
| **可执行性** | 步骤不可执行 | 部分需补信息 | 直接照做 |
| **风险识别** | 漏致命风险 | 漏非致命风险 | 全识别 |
| **边界清晰** | 范围模糊 | 部分需澄清 | 清晰 |
| **一致性** | 自相矛盾 | 局部不一致 | 一致 |

### 2.2 code/实现评审 rubric

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| **正确性** | 行为错 | 边角 case 错 | 全对 |
| **验证** | 无测试 | 测试不全 | 测试充分 |
| **范围纪律** | 越界改 | 顺手改但不影响 | 严守边界 |
| **可靠性** | 易崩溃 | 边角崩溃 | 健壮 |
| **可维护性** | 难懂 | 部分需注释 | 清晰 |
| **交接准备度** | 无法交接 | 需补充 | 可独立 ship |

### 2.3 结论判定(两套 rubric 通用)

| 结论 | 条件 |
|---|---|
| **Accept** | 全维度 ≥1 分 **且** 总分 ≥9/12 |
| **Revise** | 有 1 分项 **或** 总分 6-8 |
| **Block** | 任一关键维度为 0 **或** 总分 <6 |

## 3. 触发条件

### 3.1 双模式共同前置条件

同时满足才触发:

1. **任务满足「复杂任务」定义**(见 [`harness-router` SKILL.md](../../.agents/skills/harness-router/SKILL.md) §复杂任务判定,5 选 1):
   - 改动文件 >10
   - 涉及鉴权 / 权限 / 数据迁移 / 跨服务调用
   - plan 有 v1→v2 对抗式审查记录
   - 涉及安全敏感操作(token / 密钥 / 支付)
   - 涉及不可逆操作(删表 / 删列 / 改列类型)
2. **环境具备 ≥2 个异构模型族**(如 GLM + Claude 或 GLM + GPT)—— 当前不满足,所以不触发

### 3.2 试点期额外条件(机制未正式上线前)

- **用户主动声明「这是试点任务」** —— 试点期不自动触发,由人决定何时试

### 3.3 未来正式期额外条件

- 模式 A(`/to-spec` 阶段):plan 评审复杂任务自动触发
- 模式 B(`/code-review` 阶段):复杂任务实施完成后自动触发

## 5. 避坑设计(强化版,基于 review §4.2 共谋信号实测)

| 陷阱 | 严重度 | 缓解 |
|---|---|---|
| **共谋风险**(同家族模型相似训练数据 → 相同盲区)| 🔴 致命 | **必须用异构模型族**(GLM/Claude/GPT 不同家族);**禁止同家族多尺寸模拟**(本 plan v1 评审已验证此路不通 —— 3 票可执行性全 1 分既是真问题也是共谋信号)|
| **自举悖论**(未经验证的机制验证自己)| 🔴 致命 | 机制首次应用**必须评非机制本身的产出物**;本 plan 已删 v1 §12 强制自举设计,改为「机制落地后挑真任务试点」|
| **rubric 区分度不足**(0-2 分让 9-10 分都卡 Accept/Revise 边界)| 🟡 中 | 未来考虑 0-3 分或加权重(本 plan v1 评审暴露:9.67/12 均分达 Accept 线但因 1 分项归 Revise)|
| **成本爆炸** | 🟡 中 | 仅复杂任务触发;用户可调 N 次预算;模式 A 是融合非简单多数 |
| **仲裁者悖论**(3 票分裂时谁裁决)| 🟡 中 | 必须有 rubric 或 ground truth,**不能让仲裁者凭感觉**;模式 B 的仲裁规则:按 rubric 维度逐项重评 |
| **溯源验证缺失**(模型编事实)| 🟡 中 | 强制**至少一票是「溯源验证」**(真读源码 / 官方文档);模式 B 至少 1 个评审者必须打开实际文件核实 |
| **触发依赖 agent 自觉读文档** | 🟡 中 | 解法:harness-router 在「复杂任务评审」分支**硬提示**(见 SKILL.md),不依赖 agent 自觉 |
