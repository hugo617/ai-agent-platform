## 剧本一:周一早晨,陈馆长(朝阳店 owner)

> 账号:`chen_guanzhang` / `Demo@123456` · 体现:门店日常经营全流程

### 场景:新一周开始,陈馆长要先了解上周店里情况,然后调整本店 AI 顾问的话术。

**操作路径:**

1. **登录** → 进入朝阳店视角,首屏是 `/`(Dashboard)
   - 看到:本店客户数、本周新增、对话总数、本月 token 消耗趋势
   - 体现:**门店经营一目了然**

2. **看客户** → 左侧菜单「客户」(`/customers`)
   - 看到:张先生(颈椎理疗,vip)、刘女士(艾灸,vip)、孙阿姨(艾灸养生,vip)等
   - 顶部可用**状态筛选器**筛 vip/active/blacklist —— 演示客户 4 态管理
   - 点开张先生:看到他的全局身份 + 朝阳店档案(备注、标签、历史)
   - 体现:**客户档案 + 状态分层管理**

3. **配 Agent 话术** → 左侧「智能体」(`/agents`)
   - 找到「朝阳店健康顾问」,点编辑
   - 调整 `system_prompt`:补充「旺季优先推荐套餐组合」
   - 调推理参数:`temperature`(严谨↔发散)、`max_tokens`、`top_p`
   - 体现:**每店 Agent 可独立调参,话术随业务调整**

4. **看本月消耗** → 左侧「账单」(`/billing`)
   - 看到:本店钱包余额、累计充值、累计消耗、最近交易明细
   - 体现:**按量计费透明,每笔可追溯**

## 剧本二:李师傅接待颈椎理疗客户(朝阳店 senior_therapist)

> 账号:`li_shifu` / `Demo@123456` · 体现:AI 辅助 + 自定义权限边界

### 场景:客户张先生到店做颈椎理疗,李师傅用 AI 顾问查方案,并更新客户备注。

**操作路径:**

1. **选 Agent** → 左侧「智能体」(`/agents`)→ 选「朝阳店健康顾问」→ 点「对话」
   - 进入 `/chat`,关联到该 Agent

2. **关联客户** → 对话界面上方选择「关联客户:张先生」
   - 体现:**对话可绑定客户**,后续用量能按客户归因

3. **对话查方案**(需配置真 LLM key):
   - 输入:「张先生颈椎又有点不舒服,这次推拿+艾灸怎么安排?」
   - AI 基于朝阳店知识库(颈椎理疗操作规范)回答:建议推拿放松 + 大椎/风池艾灸,疗程频率等
   - 体现:**RAG 知识库 + LangGraph Agent**,话术符合本店标准
   - 若未配真 key:可看种子灌入的历史对话(颈椎理疗方案那段),内容一样

4. **查历史对话** → 对话列表(`/chat` 左侧)可看到本次及过往对话
   - 体现:**对话历史留存**,可回顾

5. **更新客户备注** → 回到「客户」(`/customers`)→ 张先生 → 编辑备注
   - 李师傅**能改备注**(customers:update),但**不能删客户**(无 customers:delete)
   - 体现:**自定义角色的精细权限** —— senior_therapist 正好「能改不能删」

## 剧本三:总部督导员巡查 3 门店(hq_staff)

> 账号:`hq_dudao` / `Demo@123456` · 体现:跨店只读 + 行级数据范围

### 场景:督导员要看「颐和堂中医馆」组织(朝阳+海淀)的经营情况,但不能看王府井(加盟店,另一组织)。

**操作路径:**

1. **登录** → 进入督导视角
   - 看到:朝阳 + 海淀两家店的汇总数据(客户、对话、用量)
   - **看不到王府井** —— 因为王府井属「独立养生馆」组织,督导的数据范围是 `group`(颐和堂中医馆组织)
   - 体现:**行级数据范围 group**,跨店可见但有边界

2. **跨店看客户** → 「客户」(`/customers`)
   - 看到:张先生(朝阳+海淀跨店复用)、刘女士(朝阳)等
   - 注意:张先生有跨店标识 —— 同一身份在两家店都有档案
   - 体现:**跨店客户身份复用**,督导能看出「哪些客户在多店消费」

3. **看用量分布** → Dashboard 的用量趋势
   - 对比朝阳 vs 海淀的 token 消耗,判断哪家店 AI 用得更频繁
   - 体现:**跨店数据汇总**,辅助总部决策

> 对比:用 `admin`(super_admin)登录则能看到全部 3 家店(含王府井),体现 `all` 数据范围。

## 剧本四:平台运营给门店充值(super_admin)

> 账号:`admin` / `Admin@123456` · 体现:平台级管控 + 计费配置

### 场景:新店开业,平台运营要给王府井店充值,并配置 DeepSeek 的计费单价。

**操作路径:**

1. **登录** → 进入总部视角(开发环境登录页已预填 admin)

2. **给门店充值** → 左侧「门店」(`/tenants`)
   - 找到「王府井理疗馆」,进入充值入口
   - 或走「计费管理」(`/billing/admin`)→ 选王府井店 → 充值 token 额度
   - 体现:**平台统一管控门店余额**

3. **配模型定价** → 「计费管理」(`/billing/admin`)→ 定价区
   - 配置 DeepSeek 模型单价:输入单价/1k token、输出单价/1k token
   - 这就是所有门店对话扣费的基准(平台默认价;也可为某店配租户级覆盖价)
   - 体现:**透明计费基准**,加盟商账目可核对

4. **看全局用量** → Dashboard 看全连锁用量趋势
   - 能看到 3 家店的总消耗、按 Agent/客户维度的明细
   - 体现:**全局视角**,平台运营掌控全盘成本
