核心设计原则看懂 schema 必须先懂的 5 条「宪法」

① 多租户隔离：共享数据库 + tenant_id

所有租户共库共表，租户边界靠 tenant_id 列。隔离必须在 Repository 层强制（TenantScopedRepository 自动加 WHERE tenant_id=?），不依赖 Service「记得加 where」。

跨租户越权的兜底（PG RLS）尚未上，目前靠应用层。

② 软删除 + 部分唯一索引

删除一律置 is_deleted=true + deleted_at=now()，不物理删行。唯一性用部分唯一索引（只约束未删行）实现，PG/SQLite 双方言镜像：

UNIQUE (col) WHERE is_deleted = false -- PG UNIQUE (col) WHERE is_deleted = 0 -- SQLite

查询默认带 is_deleted=False 语义。先例：uq_users_username_active、uq_users_email_active。

③ RBAC 双层结构 + SCD2 时间维度

权限分两层：显示层（roles/permissions/role_permissions）+ 执行层（casbin_rule，权威）。user_tenants 和 role_permissions 已升级为 SCD2 时态表，支持「任意时间点还原」。

历史回溯看 SCD2 表；实时鉴权看 casbin； SCD2 当前态（valid_to IS NULL）是 casbin 的同步源。 casbin 永远只回答「现在能不能」，不存历史。 写路径（业务绝不直接碰 valid_from/valid_to）： 改权限 → 写 SCD2（关旧行+插新行）→ 同步 casbin → 写 system_logs

④ 软引用（无 FK）—— 保留历史的设计

部分列故意不建外键，存 ID 但不连 FK，让用户/租户被删后业务数据仍保留用于审计。已知 3 处特例：

· conversations.user_id → users.id
· messages.tenant_id → tenants.id（冗余加速隔离查询）
· role_permissions.tenant_id → tenants.id（由 role_id 间接保证，避免级联路径冲突）

⑤ 超级管理员：跨租户 platform_role

users.platform_role 与租户内角色（user_tenants.role）正交：null=普通用户，super_admin=平台超管，鉴权时绕过所有租户权限检查。与 verification_codes 同属「平台级」而非租户级。

关联关系
多对多 · CASCADE · SCD2
tenants ↔ users（经 user_tenants）
成员关系是 SCD2 时态表：valid_to IS NULL = 当前成员。改角色走 assign_role（关旧行+插新行），保留全部历史用于「某成员当时的角色」还原。
一对多 · CASCADE
tenants → agents / conversations
租户删除时级联清除其下所有 Agent 定义与会话
一对多 · CASCADE
agents → conversations → messages
Agent 删除级联清会话；会话删除级联清消息
一对多 · CASCADE
tenants → roles / permissions
租户删除时级联清除其角色与权限定义（显示层，不影响 casbin_rule）
多对多 · CASCADE · SCD2
roles ↔ permissions（经 role_permissions）
授权关系是 SCD2 时态表：当前态（valid_to IS NULL）是 casbin 的同步源，历史行支持「某角色当时的权限集」还原。走 grant/revoke。
软关联 · 无 FK
user_tenants.role → roles.code
成员角色名与 casbin 角色名一一对应，但不建 FK（保持 casbin 解耦）
自引用 · SET NULL
organizations.parent_id → organizations.id
组织树父子关系；父组织删除时子组织父级置空（不级联）
多对多 · CASCADE
users ↔ organizations（经 user_organizations）
用户可属多个组织，is_main 标记主组织
一对多 · CASCADE
users → user_sessions / user_login_methods
用户删除时级联清除其会话与登录方式记录
一对多 · SET NULL
users / tenants → system_logs
审计日志用 SET NULL：用户/租户删除后保留历史日志（append-only 审计底座）
一对多 · CASCADE
tenants → verification_codes
验证码归属租户（可空），租户删除级联清除验证码
软引用 · 无 FK（3 处特例）
conversations.user_id / messages.tenant_id / role_permissions.tenant_id
故意不连 FK：用户/租户被删后业务数据仍保留用于审计；role_permissions.tenant_id 由 role_id 间接保证，避免级联路径冲突
